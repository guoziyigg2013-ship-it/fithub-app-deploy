# FitHub 上架前大升级与修复清单

生成日期：2026-06-24

## 当前线上入口

当前用户可访问的试用前端地址：

```text
https://fithub-cn.pages.dev/
```

当前代码配置中的 API 后端仍是：

```text
https://fithub-app-1btg.onrender.com
```

这意味着：前端已经有一个固定试用入口，但核心 API、账号数据、媒体上传和小程序接口还没有完全切到国内生产域名。正式上架前必须完成后端国内化、数据稳定化、媒体存储生产化和小程序合法域名配置。

## 一句话目标

把 FitHub 从“可演示网页试用版”升级成“账号稳定、数据不丢、页面流畅、UI 统一、微信小程序可提交审核、未来 iOS App 可继续扩展”的准生产版本。

## P0：必须先修，不修不建议上架

### 1. 国内生产域名与后端稳定化

- 确认最终正式访问域名，例如 `app.<你的域名>` 或 `www.<你的域名>`。
- 确认正式 API 域名，例如 `api.<你的域名>`。
- 后端迁移到腾讯云等国内服务器，不再依赖 Render 冷启动。
- 前端 `config.js`、小程序 `config.js`、iOS 默认服务地址统一切到正式 API。
- `/healthz` 和 `/api/storage/status` 必须稳定返回，不能出现长时间白屏或唤醒等待。

验收标准：

- 手机 Safari 打开正式域名，首屏 2 秒内可见。
- 连续 10 次打开不出现 Render 黑屏、白屏、冷启动等待。
- 小程序真机预览关闭“忽略合法域名”后仍能访问 API。

### 2. 账号与数据绝不丢

- 手机号作为主账号，一个手机号只对应一个主账号。
- 同一主账号可拥有健身爱好者、健身教练、健身房多个身份。
- 每个身份的数据环境独立：关注、粉丝、消息、预约、主页、动态、打卡记录互不串。
- 退出登录、换设备登录、更新部署后，账号、关注、消息、预约、动态都必须保留。
- 发布前后做数据快照对比，防止升级后用户或关注异常减少。

验收标准：

- 同一手机号在两台手机登录，能恢复全部身份。
- 关注 3 个对象后退出重进、换设备、重新部署，关注仍存在。
- 教练身份的预约不会跑到训练者身份。
- 训练者身份的私信不会被教练身份覆盖。

### 3. 图片、视频、头像生产化

- 头像上传必须稳定保存，刷新、退出、换设备后不消失。
- 动态图片和视频必须进入对象存储，不再依赖浏览器本地缓存。
- 图片生成缩略图，列表页优先加载缩略图，点开再看原图。
- 视频限制大小、时长和格式，上传失败要有明确提示。
- 模拟教练、模拟健身房、模拟动态使用真实压缩图片，避免演示时空白或慢加载。

验收标准：

- 上传头像后刷新 5 次仍正常显示。
- 发布图片/视频动态后，另一台手机能看到。
- 探索页图片不闪、不大片空白、不明显卡顿。

### 4. 关键操作即时反馈

- 关注、取消关注、点赞、取消点赞、收藏、取消收藏、发送私信、提交预约、提交资料都要立即反馈。
- 不允许用户点击后长时间没有反应，导致重复点击。
- 网络慢时先做本地乐观更新，再后台同步，失败时清楚提示并回滚。

验收标准：

- 关注按钮点击后 100ms 内变状态。
- 私信发送后立即出现在聊天框。
- 提交资料按钮点击后进入“提交中”状态，不能像没点一样。

## P1：上架前体验与功能统一

### 5. UI 风格统一

- 统一橙色主色、圆角、卡片、按钮、底部导航、身份标签样式。
- 首页、探索、预约、我的、小程序页面保持同一视觉语言。
- 去掉冗余按钮，例如“查看主页”改为点击头像/名字栏进入主页。
- 文案统一短句，不出现一行三个字下一行一个字的尴尬排版。

验收标准：

- iPhone 13 Pro、普通安卓、小程序预览三端不遮挡、不错位。
- 底部导航固定且不挡核心按钮。
- 同类按钮样式一致，页面不像多个 demo 拼在一起。

### 6. 小程序核心闭环

- 小程序用真实 AppID。
- 微信后台配置合法 `request`、`uploadFile`、`downloadFile` 域名。
- 小程序支持注册、登录、切换身份、关注、发动态、图片/视频、私信、预约、打卡、我的。
- 小程序隐私保护指引、用户协议、权限说明准备齐全。

验收标准：

- 微信开发者工具生产检查通过。
- 真机扫码进入后不需要关闭域名校验。
- 主要流程 30 分钟连续测试不白屏、不丢数据。

### 7. 教练与健身房端完善

- 教练端可发布可预约时间。
- 健身房端可发布可预约课程/场地时间。
- 教练和健身房在预约页看到别人给自己的订单。
- 教练和健身房的“我的”不再像训练者页面的空壳，要有订单、预约、评价、粉丝、消息、主页管理。

验收标准：

- 教练发布时间后，训练者能预约。
- 训练者预约后，教练端能看到待确认订单。
- 健身房能维护价格、场馆照片、可预约服务。

### 8. 消息、私信、通知

- 消息按身份独立。
- 私信列表像微信一样以头像为主，有未读红点。
- 点击头像或名字进入对方主页。
- 对方主页支持关注、私信、拉黑。
- 被拉黑后不能继续私信或互动。

验收标准：

- A 给 B 私信，B 当前身份收到未读数。
- 切换到另一个身份，未读数不串。
- 拉黑后消息发送被阻止并提示。

## P2：审核与运营能力

### 9. 内容安全与审核后台

- 动态、评论、私信、头像、图片、视频接内容安全。
- 用户可举报。
- 管理员可处理举报、删除内容、封禁账号。
- 小程序审核前准备测试账号和审核说明。

验收标准：

- 敏感文本能进入审核队列。
- 举报后管理员后台能看到。
- 被封禁账号不能继续发布。

### 10. 正式测试与回归自动化

- 建立上架前回归测试清单。
- 自动测试覆盖账号、关注、动态、媒体、预约、消息、打卡、健康、商城、权限。
- 每次部署自动跑核心 smoke。
- 发布后生成证据报告：域名、API、存储、媒体、数据快照、关键流程结果。

验收标准：

- 一键执行核心回归测试。
- 测试失败不发布。
- 发布前后数据计数不异常下降。

## 建议执行顺序

1. 固定正式域名与 API：确认国内域名、腾讯云后端、HTTPS、前后端配置。
2. 数据稳定化：账号、身份隔离、关注/消息/预约不丢，发布前后快照对比。
3. 媒体系统：头像、图片、视频对象存储、缩略图、真实图片性能优化。
4. 交互即时反馈：关注、点赞、收藏、私信、预约、资料提交全部乐观更新。
5. UI 统一：网页和小程序按同一套设计语言整理。
6. 小程序闭环：真实 AppID、合法域名、登录注册、发布、预约、我的。
7. 教练/健身房端：可预约时间、订单、评价、主页管理。
8. 审核合规：隐私政策、用户协议、内容安全、举报后台、账号注销。
9. 全量回归：真机、多身份、多终端、发布后数据不丢。
10. 提审准备：小程序审核材料、测试账号、服务类目和权限说明。

## 我建议本次大升级的第一步

先做“生产链路稳定化”，也就是：

- 把当前 `https://fithub-cn.pages.dev/` 的 API 从 Render 切到国内正式 API。
- 完成腾讯云后端、HTTPS、健康检查、数据快照、发布证据报告。
- 确认升级后老用户、新用户、关注、消息、头像、动态都不会丢。

原因很简单：UI 可以慢慢精修，但账号和数据如果不稳，后面所有功能都会被反复打断。
